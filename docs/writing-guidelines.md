## 振り返りフレームワーク: YWT

出典: 日本の経験学習フレームワーク。KPTよりimmutable logとの構造的親和性が高い。

**選定理由（2026-04-04 調査）:**
- YWTの「Y（やったこと）」がimmutable logと直接対応する（KPTには事実記録の層がない）
- 個人の研究プロジェクトに最適（KPTはチーム合意形成向き）
- 「W（わかったこと）」が気づき・失敗・構造的欠陥を捕捉する自然な場所になる

### 構造

| 項目 | 内容 | 例 |
|------|------|-----|
| **Y（やったこと）** | 事実の記録。何を実装/実行/分析したか | signal_runner.py実装、31テスト |
| **W（わかったこと）** | 気づき・学び・発見。特に驚きや失敗 | shuffle p値の離散制限でBHが機能しない |
| **W-lack（不足していたもの）** | 構造的欠陥。事前に用意すべきだったもの | float信号の反証テスト動作検証が未実施 |
| **T（つぎにやること）** | 次セッションへの引き継ぎ | Stage 3実装、cat 49探索 |

### W-lack の追加

4Lsフレームワーク（Liked/Learned/Lacked/Longed for）から「Lacked」をYWTに補強。
構造的欠陥の記録漏れを防ぐ目的。

### 適用場所

- PR #1 のコメント（セッション単位、immutable）
- WIP-progress.md のワークログ

## 認知負荷最小化の7原則

出典: Sweller認知負荷理論 + UX Tigers調査 + zakirullin/cognitive-load (GitHub 7k+)
調査日: 2026-04-04

AI生成テキストの最大問題は「正しいが冗長」。以下を適用する。

### 原則

1. **結論先行**: 最も重要な情報を最初に（逆ピラミッド構造）
2. **Why > What**: コードを見れば分かることは書かない。判断の理由を書く
3. **1文1概念**: 日本語は助詞連鎖で文が長大化しやすい。短文化
4. **段階的開示**: サマリー→詳細の2層。全部読まなくても把握できる状態
5. **冗長な構造化を避ける**: 表・箇条書きは情報量に見合うときだけ使う
6. **具体的な数値で語る**: 形容詞（「多くの」「大幅に」）を排除
7. **読者の行動を想定する**: 「で、次に何をすればいい？」に答える

### アンチパターン（避けるべき）

- 冗長な前置き（「まず〜を確認し、次に〜を調査し...」）
- 過剰な表（3行以下のデータに表は使わない）
- 同じ情報の重複（サマリーで書いたことを詳細で繰り返す）
- コードに書いてあることのドキュメント化
