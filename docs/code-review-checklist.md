# Code Review Checklist: 統計・金融コード

導入根拠: ADR-022
Created: 2026-04-02

本プロジェクトの統計検定・金融データ処理コードをレビューする際の基準。
一般的なコード品質（命名・構造・テストカバレッジ）は対象外。ドメイン固有の落とし穴に絞る。

## 1. 統計的正確性

### 並替テスト（Permutation Test）
- [ ] p値に +1 補正があるか: `(count + 1) / (n_perms + 1)` — p=0.0 は不正（Phipson & Smyth 2010）
- [ ] シャッフル対象が正しいか（シグナルのみ? リターンのみ?）。意図しない相関の破壊がないか
- [ ] 定数シグナル（全True等）でシャッフルがno-opになるケースを検出・警告しているか
- [ ] rng_seed が再現性のために公開されているか

### 多重検定補正（BH法）
- [ ] p値をソートしてからBH手順を適用しているか
- [ ] 最大の k（p(k) <= threshold(k)）を見つけ、k以下を全て棄却しているか
- [ ] 結果が元の入力順序に正しくマッピングされているか
- [ ] FDR制御（期待偽発見割合）とFWER制御（族エラー率）を混同していないか

### シグナル検定
- [ ] bool/float の型判定が正確か。float 0.0/1.0 が bool と誤判定されないか
- [ ] 二項検定・Spearman相関の片側/両側が意図通りか
- [ ] N<30 のフィルタが全レイヤーで一貫しているか（事前チェックと内部チェックの乖離がないか）

## 2. 金融データ処理

### スプリット調整
- [ ] 調整係数 = AdjClose / Close で算出しているか
- [ ] OHLC に係数を乗じ、Volume は係数で割っているか（逆方向）
- [ ] 日足index(naive) と 5分足index(tz-aware) の型不一致を正しく処理しているか
- [ ] 日足データが存在しないシンボルで係数=1.0（無調整）にフォールバックしているか
- [ ] 複数スプリットが累積するケースで係数が正しいか

### Look-Ahead Bias
- [ ] 時刻 t で使うデータが t 時点で利用可能か
- [ ] Walk-forward の分割が時系列順で非重複か
- [ ] レジームラベルが未来の情報を含んでいないか

### セッション区分
- [ ] レギュラーセッション(09:30-15:55 ET)のフィルタが正しいか
- [ ] プレマーケット/アフターマーケットの含有が意図通りか
- [ ] DST切替日でバー数が変わるケースへの対応

## 3. 並行処理・I/O

### ファイルロック
- [ ] 競合するリソースアクセスにロック（fcntl.flock等）があるか
- [ ] file_exists チェックがロック取得**後**にあるか（TOCTOU防止）
- [ ] CSV のカラムスキーマが全エージェント間で一致するか。不一致時の挙動が定義されているか

### 再現性
- [ ] 乱数生成に seed パラメータがあるか
- [ ] テストで同一 seed が同一結果を保証するか
- [ ] 浮動小数点の比較に適切な許容誤差（pytest.approx等）を使っているか

## 4. テスト設計

テスト設計の詳細原則は @docs/testing-guidelines.md を参照。

- [ ] テスト関数にassert文が1つ以上存在するか（assert欠落テストは常にパスする。session 9で発見）
- [ ] テスト名が assert の内容と一致しているか（`test_X_fails` なら `passed is False` を検証）
- [ ] 既知解・境界・不変量・反例の4層が揃っているか
- [ ] 境界テストは決定論的データ（ランダム性なし）で構築されているか
- [ ] テスト内のマジックナンバーに根拠コメントがあるか

## 更新ルール

- 新たなドメイン固有バグを発見したら、該当セクションに項目を追加する
- 一般的なコード品質（PEP 8、型ヒント等）は追加しない
- 各項目に根拠（論文・事例）を付記する
