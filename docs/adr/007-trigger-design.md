# ADR-007: 成長メカニズムのトリガー設計

Status: accepted
Date: 2026-03-27

## Context

Charterに定義された成長メカニズム（Brier score、バイアス検出、知見検証、事後検証等）は
コードとして存在するが、それを起動するトリガーがない。
部品はあるが回路として接続されていない。

Claude Codeには3つのトリガー機構がある:
- **Hooks**: ライフサイクルイベントで確定的に実行。stdoutがコンテキストに注入される
- **CLAUDE.md**: セッション開始時に読み込まれる持続的コンテキスト。実行ではなく指針
- **Skills**: ユーザーまたはClaudeが明示的に呼び出すワークフロー

## Decision

> **3層のトリガー設計で成長メカニズムを接続する。**

### 層1: SessionStart Hook（自動・確定的）

セッション開始時にPythonスクリプトを実行し、状態をコンテキストに注入。

**注入する情報:**
- 期限切れ/期限間近の予測（解決を促す）
- 最新レジーム判定（データの鮮度を含む）
- stale知見の警告（180日未検証）
- データの最終更新日時

**実装:** `.claude/hooks/session_start.py` → stdout → コンテキスト注入

### 層2: CLAUDE.md ルール（行動指針）

Hookが「何が起きているか」を伝え、CLAUDE.mdが「何をすべきか」を指示する。

**ルール:**
- エントリー分析を行ったら、予測を起草してユーザーに提案する
- 予測の期限が過ぎていたら、解決を最優先で実行する
- 市場で驚いたことがあれば、知見として記録を提案する
- セッション中にレジーム判定に影響するデータを得たら、market_observationsに記録する

### 層3: Stop Hook（セッション終了時チェック）

セッション終了前にprompt型フックで確認。

**チェック項目:**
- condition.mdが最終更新日と乖離していないか
- 未記録の予測や知見がないか

## Rationale

- **SessionStart hook**: Claude Code公式推奨。stdoutがコンテキストに注入される確定的メカニズム
- **CLAUDE.md**: 行動指針として公式推奨。ただし実行を保証しない（Hookと組み合わせて補完）
- **Stop hook (prompt型)**: 軽量モデル（haiku）で判定。セッション終了漏れを防ぐ
- **Hookでやらないこと**: DB書き込み、API呼び出し等の副作用。Hookは「状態の表示」のみ。実際の記録はClaudeがセッション中に行う

## Consequences

- .claude/hooks/session_start.py を作成
- .claude/settings.json にHook設定を追記
- CLAUDE.md にトリガールールを追記
- 成長メカニズムの起動が受動的→能動的になる
