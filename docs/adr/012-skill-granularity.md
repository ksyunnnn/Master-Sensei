# ADR-012: スキル粒度設計の原則

Status: accepted
Date: 2026-03-30

## Context

4つのスキル（scan-market, update-regime, review-events, verify-knowledge）と `update_data.py` の関係が暗黙的で、日次ワークフローが断片化していた。

具体的な問題:
1. `/scan-market` と `/update-regime` は実運用でほぼ毎回セットで実行されるが、独立設計
2. `update_data.py` がスキルでもHookでもなく手動コマンドとして浮いている
3. ユーザーの「市場を判断したい」が3ステップ（scan-market → update_data.py → update-regime）に分散
4. スキル間の粒度・分割基準が未定義（ADR-008はHook/Skill/CLAUDE.mdの責務分担のみ）

Claude Code公式ガイドラインおよび既存ADR（007, 008, 010）を調査し、スキル粒度の設計原則を定めて上記問題を解決する。

## Options

| 選択肢 | 長所 | 短所 | 採否 |
|--------|------|------|------|
| A: スキルを統合（scan-market + update-regime → 1スキル） | 1コマンドで完結 | 単独実行不可、責務混在、500行超の懸念 | 不採用 |
| B: 原則を定めずケースバイケース | 柔軟 | 判断基準なし、同じ議論を繰り返す | 不採用 |
| C: 粒度原則を定義し、オーケストレーション層で統合 | 各スキル独立性維持、ワークフロー明確化 | CLAUDE.mdの行動ルール追加が必要 | 採用 |

## Decision

> スキルの粒度は以下の5原則に従う。スキル間の調整はCLAUDE.mdの行動ルールで定義する。

### 原則

#### P1: 単一責務 — 1スキル = 1つの判断サイクル

スキルは「入力→判断→出力」の1サイクルを完結させる。異なる判断コンテキスト（異なる情報源、異なる専門性）が必要なら別スキル。

現在の適用:
- `/scan-market`: ニュース→重要度判断→イベント登録
- `/update-regime`: マクロ指標→レジーム判断→記録
- `/review-events`: 過去イベント→事後検証判断→lesson記録
- `/verify-knowledge`: 既存知見→有効性判断→検証日更新

#### P2: 単独実行可能性 — どのスキルも単独で意味のある結果を返す

スキルAを実行するためにスキルBの事前実行が必須、という依存は作らない。前提条件（データ鮮度等）が未達の場合、スキルは**警告して続行**または**スキル内で自己解決**する。

#### P3: インフラと判断の分離 — データ取得はスキルが所有しない

API呼び出し・Parquet更新等のインフラ処理は、スキルの外（`update_data.py`等）に置く。スキルは前提条件としてインフラを**呼び出す**ことは許容するが、インフラのロジック自体は**所有しない**。

現在の適用:
- `/update-regime` → step 1で `update_data.py --macro-only` を呼ぶ（許容: 呼び出しのみ）
- `update_data.py` のProviderChainロジックはスキル外に存在（所有しない）

#### P4: オーケストレーションはスキルの外に置く

スキル間の実行順序・組み合わせはスキル自体に書かない。CLAUDE.mdの行動ルールまたはSessionStartフックの状態注入で調整する。

根拠: Claude Code公式 — スキル同士は直接呼び出せない。合成はCLAUDE.mdまたはHookで行う。

#### P5: 状態検出と行動指示の分離（ADR-007踏襲）

Hookが「何が起きているか」を検出し、CLAUDE.mdが「何をすべきか」を指示し、スキルが「どうやるか」を定義する。

### 日次ワークフローの定義（P4の適用）

SessionStartフックが注入する状態に基づき、CLAUDE.mdに以下の行動ルールを追加する:

```
データ鮮度が1日以上古い場合 → update_data.py を実行してからレジーム判定を提案
```

これにより `/scan-market` → `update_data.py` → `/update-regime` の順序がCLAUDE.mdの行動ルールとして明文化される。スキル自体は変更しない。

## Rationale

- Claude Code公式: スキルは独立したライフサイクルを持つなら分割を維持する
- Claude Code公式: スキル間のオーケストレーションはCLAUDE.mdで行う（スキル同士は直接呼べない）
- ADR-007: Hookは状態検出、CLAUDE.mdは行動指示、Skillは実行手段
- ADR-008: 緊急度で自動/手動を分離する原則はスキル内部の粒度にも適用可能
- Charter 3.5: 5原則で説明できるなら7つ使わない

## Consequences

- CLAUDE.mdに日次ワークフローの行動ルールを追加
- 既存4スキルの構造は変更しない
- `update_data.py` はインフラとしての位置づけを明確化（スキルにはしない）
- 見直しトリガー:
  - スキル数が6を超えたとき
  - 同じスキルの組が3セッション連続で一緒に呼ばれたとき
